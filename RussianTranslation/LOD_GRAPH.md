## What changed, and why

The prior export ([`src/export_interop.py`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/export_interop.py) →
[`release/ontolex.ttl`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/release/ontolex.ttl))
is a **flat one-way string dump**: a placeholder `example.org` IRI namespace,
`ontolex:usage` as a bare string, one synthetic `pwg:sourceRef` literal per
sense, and **no** sense-relations, evidence grades, resolvable citations, dating,
or query surface. It satisfies "we emit OntoLex" on paper and nothing a
lexicographer would actually query.

The new generator, [`src/export_lod.py`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/export_lod.py),
upgrades it in place (it does **not** replace `export_interop.py`, which still
produces the TEI Lex-0 + reverse-index artifacts) and adds the five things a real
LOD graph needs:

1. **Real, dereferenceable, configurable IRIs** — `--base-iri`, defaulting to the
   documented placeholder `https://w3id.org/sanskrit-lexicon/pwg-ru/` pending the
   publication-domain ruling (see *Open decision* below). Instance resources are
   absolute `<base+path>` IRIs; vocabulary terms are the `pwglex:`/`gr:` prefixes.
2. **`vartrans` sense↔sense relations** from
   [`pwg_ru_relationships.jsonl`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/pwg_ru_relationships.jsonl)
   — `restate`/`add`/`relocate`/`correct`, `abridging`/`additive`, per source
   layer (`pw`/`pwkvn`/`sch`/`nws`), with the LLM confidence + evidence string.
3. **PROV-O evidence grades** — every sense carries `pwglex:evidenceGrade` →
   a SKOS concept (`gr:approved`, `gr:human-reviewed`, `gr:machine-preview`,
   `gr:dict-attested`, `gr:kow`, `gr:corpus`), each flagged `pwglex:citable`. The
   **citable release is now a SPARQL filter**, not a separate build — machine
   translations stay in the graph but graded non-citable, exactly as the G5 gate
   in `export_interop.py` intends.
4. **Per-sense `<ls>` citations as first-class resources** — each `<ls>` becomes a
   `pwglex:Citation`/`prov:Entity` with `pwglex:sourceSigla`, `pwglex:locus`, and
   the raw text, linked `dct:references` from the sense.
5. **Renou stratum as dated attestation** from
   [`pwg_sense_stratum.jsonl`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/pwg_sense_stratum.jsonl)
   — `pwglex:StratumAttestation` with Renou oldest/youngest, `date_min`/`date_max`,
   dated-citation count, and the Russian stratum label.

**The lemma bank / LiLa link.** One shared `ontolex:Form`/`lila:Lemma` node per
`key1` (the SLP1 spine) is the join hub: every homograph `ontolex:LexicalEntry`
points at it via `ontolex:canonicalForm`, and the **separate** DCS-frequency graph
([`dcs_freq.ttl`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/release/fixture/dcs_freq.ttl))
attaches `pwglex:dcsCount`/`dcsBand` to the *same* lemma IRI. This mirrors
[LiLa's lemma-bank pattern](https://openhumanitiesdata.metajnl.com/articles/10.5334/johd.145)
and the [Lexicala→LiLa 2025](https://aclanthology.org/2025.ldk-1.21/)
interlinking method: the lemma is the interoperability anchor, and cross-dataset
queries join on it. (There is no published Sanskrit LiLa endpoint to `owl:sameAs`
against yet; the local lemma node is modelled `lila:Lemma` so a future
`owl:sameAs` to a real Sanskrit lemma bank is a one-line add.)

## Known modelling boundaries (honest, not padded)

- **Homograph resolution.** The assembled cards carry no homonym key, so all
  translation senses for a `key1` hang under its **first** entry — the same
  discipline `export_interop.py` already applies. A per-homonym entry split needs
  a homonym→homograph map that does not exist in the source data.
- **vartrans target = entry, not sense.** `insertion_point` names a target
  `key1` + homonym + sense number, but a stable per-(homonym, sense) sense IRI
  does not exist in the current backbone. The relation therefore targets the
  **entry** for that `key1`, with `pwglex:targetHomonym` / `pwglex:targetSenseTag`
  recorded as literals. Precise sense→sense edges land when the homograph split does.
- **Stratum join is on numeric `sense_tag`.** Non-numeric tags (`caus`, `desid`, …)
  carry no `pwg_sense_stratum.jsonl` row and get no attestation.

## Acceptance

[`src/lod_acceptance.py`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/lod_acceptance.py)
is the gate (exit ≠ 0 on any failure):

- **A. Federated join** — [`release/query/sense_citation_dcsfreq.rq`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/release/query/sense_citation_dcsfreq.rq)
  joins a PWG sense → its `<ls>` citation (lexical graph) **and** its lemma's DCS
  frequency (frequency graph) across the shared lemma IRI. Must return rows.
- **B. Lossless round-trip** (the re-glue byte-identity discipline, applied to RDF):
  **B1** regeneration is byte-identical; **B2** parse→serialise→parse is
  graph-isomorphic; **B3** every `<ls>`, stratum, and evidence grade in the
  **source** (recounted from the jsonl over the fixture keys) survives as a triple.
- Structural invariants + **SHACL** conformance ([`release/shapes.ttl`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/release/shapes.ttl), when `pyshacl` is present).

Fixture-only mode (no gitignored source) runs A + B2 + invariants + SHACL, so it
is CI-safe. Committed fixture: [`release/fixture/`](https://github.com/gasyoun/SanskritLexicography/tree/master/RussianTranslation/release/fixture)
over `rakz` (a verb root exercising all five constructs) + `a`/`aMSa`/`aMSaka`
(4157 triples: 12 entries, 97 senses, 372 citations, 40 relations, 2 strata).

```
# regenerate the fixture (needs the gitignored src/*.jsonl):
python src/export_lod.py lexicon  --keys-file release/fixture/fixture.keys --generated-at 2026-07-08 --out-dir release/fixture
python src/export_lod.py dcs-freq --keys-file release/fixture/fixture.keys --generated-at 2026-07-08 --out-dir release/fixture
# full graph (large):  python src/export_lod.py all --out-dir release
python src/lod_acceptance.py       # acceptance gate
```

## Boundary: who owns what

Per the 2026-06-03 four-repo split
([`ROADMAP_ATLAS_FAIR_PUBLICATIONS_2026_2027.md`](https://github.com/gasyoun/SanskritLexicography/blob/master/ROADMAP_ATLAS_FAIR_PUBLICATIONS_2026_2027.md)
§4 / G2), OntoLex/LOD
**exports** live in [`csl-standards`](https://github.com/sanskrit-lexicon/csl-standards),
not here. The routing contract for this build:

- **`RussianTranslation` (this repo)** owns the **generator + input data +
  acceptance + fixture** — the graph is derived from the gitignored pipeline
  stores, which stay here.
- **`csl-standards`** owns the **published graph + the SPARQL surface** (the
  query, the SHACL shapes, the graph-model doc). Full publication of the graph
  **data** is gated on G5 sense approvals (today the store is 100 %
  `ai_translated`, i.e. `gr:machine-preview` / non-citable) and a
  [`/publish-safety-check`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation) pass —
  see [`csl-standards`](https://github.com/sanskrit-lexicon/csl-standards) for the landed surface.

## Open decision

- **Publication IRI domain (`@DECIDE`).** IRIs are permanent identifiers, so the
  real dereferenceable domain is a human ruling, not a code default. The generator
  ships `https://w3id.org/sanskrit-lexicon/pwg-ru/` as a **placeholder** and takes
  `--base-iri`; nothing downstream hard-codes it. Registering a w3id PURL (or
  choosing a `samskrte.ru` / project domain) is mirrored to
  [`Uprava/GTD_NEXT_ACTIONS.md`](https://github.com/gasyoun/Uprava/blob/main/GTD_NEXT_ACTIONS.md).
