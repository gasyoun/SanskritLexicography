# PWG-RU launch failure ledger

_Created: 07-07-2026 · Owner: every session that launches Workflow/API work_

This is the canonical per-launch failure ledger for `RussianTranslation`.
[`PIPELINE_HISTORY.md`](PIPELINE_HISTORY.md) is the curated narrative map; this
file is the operational incident register. Every Workflow/API launch that hits a
failure, drift, retry, kill, stale artifact, cost blow-up, or suspicious
residual gets one entry here before the handoff is closed.

## Closeout rule

A launch handoff is not done until each failure is classified as one of the
categories below and has a disposition: fixed, structurally guarded,
accepted-as-proven-residual, or routed to a new bug-hunt handoff. Do not leave a
second occurrence of the same `unknown` shape in this ledger; the second
occurrence requires a bug-hunt handoff.

Run the check before closing launch work:

```powershell
python src\pilot\check_launch_ledger.py --handoff H220
python src\pilot\check_launch_ledger.py --since 2026-07-05
```

## Statistics & methodology (what makes this a paper, not a diary)

The typology below is the *narrative* register. Its quantitative companion is
[`LAUNCH_STATS.md`](LAUNCH_STATS.md), auto-generated by
[`src/pilot/harvest_launch_stats.py`](src/pilot/harvest_launch_stats.py) from the
window ledger + this file + the content-failure gallery. Re-run it before
closing launch work; never hand-write the numbers.

Four methodology rules, adopted 08-07-2026 (H390) so the ledger yields an
evidence table instead of an anecdote list:

1. **Report every incident against the population denominator.** An incident is
   not closed until its rate is stated against the launched-window count in
   `LAUNCH_STATS.md`. As of 08-07-2026 that population is 450 windows / 55 roots;
   the honest hard-fail rate is 25% (stale-refused + crashed), and **`sTA` alone
   is 98 of 450 windows** — the giant polysemous roots dominate the failure mass,
   which is the paper's central finding, not a footnote.

2. **`needs_requeue` is not a failure.** It is the normal iterative state (329/450
   windows). Folding it into "failed" is the denominator error this whole
   apparatus exists to prevent. Hard-fail = `stale-refused` ∪ `crashed` only.

3. **Three failure axes, kept separate — do not merge them.**
   - *Orchestration* — this file's 11-class `classification` (the harness broke).
   - *Content/correctness* — [`failures/failures.jsonl`](failures/failures.jsonl)
     `mode` (the translation/data was wrong: fabrication, off-by-one, …).
   - *Operational state* — the ledger's `state` (clean / partial / iterating /
     stale-refused / crashed).
   The paper presents these as distinct axes; a fabrication bug and a 429 cliff
   are not the same kind of failure.

4. **Instrument per-window model + cost, or the model question stays unanswerable.**
   The ledger currently stamps output-tokens on 1/450 windows and **no generation
   model at all**, so "does Fable reduce failures?" cannot be computed — the 6/7
   Sonnet incidents merely reflect Sonnet being the default generator. The
   standing Fable A/B (H390) requires: (a) every `window_ledger` row stamps
   `gen_model` + token split + wall-clock; (b) matched windows (same roots, same
   lane) run with model as the *only* variable; (c) report hard-fail rate,
   `structured-output-limit` rate, and $/clean-card per model. Until (a) lands,
   back-fill cost from retained transcript dirs via
   [`src/pilot/parse_workflow_cost.py`](src/pilot/parse_workflow_cost.py).

## Failure typology

- `concurrency/api`: rate limits, connection loss, mid-stream stalls, provider
  throttling, transient API nulls.
- `structured-output-limit`: model cannot emit the required schema because the
  whole-card/whole-batch output is too large or complex.
- `complexity-estimate-drift`: preflight or cost estimate misses real agent,
  token, or fragment fan-out.
- `kill-gate-calibration`: timeout budget is too loose, too strict, or lacks a
  fallback lane for this launch shape.
- `gate-bug`: deterministic audit or semantic-risk gate misclassifies a valid
  or invalid card.
- `artifact/provenance`: stale output, rootmap/input hash mismatch, ordering
  bug, missing SHA/provenance, or manual-save loss.
- `tm/cache`: translation memory or cache serves content that should have been
  regenerated or rejected.
- `filesystem/watcher`: untracked/gitignored output is deleted or overwritten
  by local repo automation.
- `key/schema-mismatch`: valid model output is dropped because keys, safe names,
  wrappers, or schema paths disagree.
- `operator/process`: concurrency discipline, claim/registry drift, manual
  capture, or handoff procedure fails.
- `external-api`: non-Workflow service quirks such as JSON-mode requirements,
  escaped-character parsing, or retriable DeepSeek failures.
- `unknown`: not yet explained. A repeated `unknown` shape must be escalated to
  a bug-hunt handoff.
