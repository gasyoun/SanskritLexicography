## 1. Why a bake-off, not a chosen rubric

pwg_ru's QA to date is deterministic and *structural* (coverage, sense-dupe, markup
fidelity, NWS owner-map — see
[`RUN_FREQ_MAX.md`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/pilot/RUN_FREQ_MAX.md));
there has been **zero reference-based or human evaluation**. An ACL-Anthology reviewer
would require a held-out human-judged sample with a defined rubric, inter-annotator
agreement, and an automatic-metric signal. MG's ruling (interview 05-07-2026): do not
pre-pick a rubric — run all four standard paradigms on the SAME sample and let
reliability / discrimination / cost / diagnostic value choose the standing protocol.
The bake-off itself is a citable methodological result ("which human-eval protocol fits
LLM-assisted dictionary translation").

## 2. Sample (held-out, 0 new Claude generation)

Built by [`src/h178_eval_bakeoff.py`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/src/h178_eval_bakeoff.py)
`sample` (seed 178, deterministic): **30 already-promoted glosses across 22 roots**
(handoff floor: ≥3), stratified as

- **20 random** sense rows (≤4 per root, forcing spread), and
- **10 flagged** — a deliberate oversample from the 16 promoted-as-is keys the
  [H178 re-audit](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/pwg_ru/H178_REAUDIT_2026-07-06.md)
  surfaced, converting an untrusted forensic signal into human verdicts.

The sample, all channel outputs, and the sheets embed unpublished `ru` text, so they
are **gitignored** (`pwg_ru/eval/`, `review/h178_*_sheet.html`); the repo is public.
Regenerate with the commands in §6.

## 3. The four rubrics (one HTML voting sheet each, org /review-sheet convention)

All four run on the same 30 items; sheets are blinded/randomized where applicable.
Markdown checkboxes are banned; votes export as `decisions.json`
(`{sheet_id, generated, decided, items:[{id, decision, note}]}`) with rubric widget
values auto-serialized into `note` as an `H178:{...}` JSON line.

| sheet | paradigm | annotator action | decision semantics |
|---|---|---|---|
| `h178_mqm` | **MQM-adapted error annotation** (Lommel et al. 2014; Freitag et al. 2021, [TACL](https://aclanthology.org/2021.tacl-1.87/)) | severity per category: mistranslation · omission · addition · terminology · fluency (none/minor/major/critical) + worst error span in the note | approve = no errors / reject = errors marked |
| `h178_likert` | **Adequacy/fluency Likert 1–5** (classic LDC-style two-scale judgment) | two 1–5 selects | approve = acceptable as-is |
| `h178_da` | **Direct Assessment 0–100** (Graham et al. 2013, [LAW 7](https://aclanthology.org/W13-2305/) — verified) | one slider | approve = publishable |
| `h178_pairwise` | **Pairwise ranking** (WMT tradition, Callison-Burch et al. 2007, [W07-0718](https://aclanthology.org/W07-0718/) — verified) | compare candidate A vs B, per-item randomized blinding | approve = A better / reject = B better / defer = tie |

The MQM category set deliberately mirrors the deterministic gates
(coverage→omission, sense-dupe→addition, markup→fluency/formatting, NWS
owner-map→terminology/attribution), so MQM annotations feed the
[`ERROR_ANALYSIS.md`](https://github.com/gasyoun/SanskritLexicography/blob/master/RussianTranslation/pwg_ru/ERROR_ANALYSIS.md)
taxonomy (B-4) directly.

**Pairwise arm B** is a **DeepSeek (`deepseek-chat`) baseline translation** of the same
German source (markup-preserving prompt) — the "cheap non-Claude baseline" of the
handoff. It is generated once, gitignored, and blinded per item with a seeded RNG
stream shared between the sheet builder and the model channel (selftest-pinned).

## 4. Annotation channels — honest labels

- **Human channel: MG, single pass** (single annotator; recruiting a second is
  deferred for 2026 by standing ruling — do NOT open a GTD item for it). Optionally an
  MG test–retest pass later for an intra-annotator figure.
- **Model channel: DeepSeek `deepseek-chat`** — the org's non-Claude backend, chosen to
  sidestep the LLM-judge self-preference/circularity problem (generator and session
  orchestrator are both Claude). One strict-JSON verdict per item covering all four
  rubrics. **Every agreement figure is therefore human×model (or model×model); none may
  ever be relabeled human×human.**
- **Automatic channel: COMETKiwi** reference-free QE — see §5.

## 5. Automatic metric — COMETKiwi (secondary; currently BLOCKED on a human step)

Reference-free QE per MG's ruling: no human reference translation exists. Planned model
**`Unbabel/wmt22-cometkiwi-da`** (0.5B, CPU-feasible; the XL variant
`Unbabel/wmt23-cometkiwi-da-xl` needs a GPU box) — WMT22 QE shared-task winner line
(Rei et al. 2022, [WMT22, aclanthology.org/2022.wmt-1.60](https://aclanthology.org/2022.wmt-1.60/) —
verified; XL: Rei et al. 2023, [aclanthology.org/2023.wmt-1.73](https://aclanthology.org/2023.wmt-1.73/);
[COMET repo](https://github.com/Unbabel/COMET)).

**Blocked**: the checkpoints are gated on Hugging Face behind a license acceptance +
account token, and this machine has no HF token. One-time human steps:

1. `pip install unbabel-comet`
2. `huggingface-cli login` and accept the license at
   [huggingface.co/Unbabel/wmt22-cometkiwi-da](https://huggingface.co/Unbabel/wmt22-cometkiwi-da)
3. `python src/h178_eval_bakeoff.py comet`

**Caveat (carry into any report verbatim):** COMETKiwi is trained on modern
sentence-level MT and is **unvalidated on 19th-century scholarly German → modern
scholarly Russian dictionary glosses** with embedded SLP1/markup — it is a
triangulation aid, not a headline metric. Its value here is exactly the correlation
test: `compute` reports Spearman ρ of COMETKiwi against each human rubric on the shared
sample, which both validates whether an automatic metric is usable at all in this
domain and helps adjudicate the winning rubric.

## 7. Scoring the bake-off (what `compute` reports per rubric)

| axis | operationalization |
|---|---|
| **reliability** | Cohen's κ human×model on binarized verdicts (MQM: any major+ error; pairwise: A/B/tie) and Spearman ρ human×model on scalar rubrics (Likert adequacy, DA) |
| **discrimination** | SD of the human scores across the 30 items + flagged-vs-random separation (does the rubric distinguish the oversampled suspected-weak cards?) |
| **annotation cost** | MG's wall-clock minutes per sheet (noted at vote time; the export contract carries decisions only, so cost is logged manually) |
| **diagnostic value** | qualitative: MQM localizes error class+span; Likert/DA localize nothing; pairwise localizes only relative preference. Scored in the protocol summary against the B-4 taxonomy's needs |

The **standing protocol is chosen after MG's votes** by: highest reliability subject to
acceptable cost, breaking ties by diagnostic value. The decision lands in this file
(§8) with the filled bake-off table, and the winning rubric becomes the recurring QA
instrument (e.g. per-window judge-sample replacement or complement).

## 9. Known limitations of this pilot

- n=30 is a pilot size: κ/ρ confidence intervals will be wide; the bake-off picks a
  *protocol*, it does not certify store quality.
- Single human annotator: no human×human agreement is measurable (standing 2026
  deferral); the model channel is a stand-in with its own biases (DeepSeek may share
  LLM-typical blind spots with the Sonnet generator even without self-preference).
- The DeepSeek pairwise baseline is a *cheap* baseline, not a competitive Sanskrit-aware
  system; a pairwise win for the store is necessary, not sufficient.
- The flagged stratum is intentionally biased toward suspected-weak cards; whole-store
  quality extrapolation must weight the strata accordingly.
