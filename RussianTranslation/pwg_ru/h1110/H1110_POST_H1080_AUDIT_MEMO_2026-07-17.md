The pre-stamp backup **is** the handoff's expected baseline. Marker counts match #517's report
exactly (`h_reconstructed` 468, `iast_reconstructed` 462, `grammar_defaulted_empty` 468), and an
independent positional diff of live-store vs. pre-stamp backup shows `h` changed: 0,
`generator` changed: 0, `generated_at` changed: 0.

**Ruling — recorded as a lineage, not drift:** the current canonical store is
**`763b844b…deb40d` (post-#517)**; its immediate predecessor is **`f15caf7d…77b2f9` (the pre-hstamp
backup)**, which is byte-for-byte the hash the handoff pinned. The only change between them is the
additive #517 markers (proven above); `h`/`generator`/`generated_at` are untouched. This is an
explained, owner-authorised lineage step — not the "unexpected store drift" that blocks paid execution.
Both hashes are recorded so the lineage is auditable.

## 3. Phase 6 — named stop: `STOP_PROFILE_UNBINDABLE`

**The paid c4 ladder was not entered, and no paid call was made.** This is a measured stop, not
an omission.

Phase 6 requires every call to carry an exact c4 profile fingerprint ("c4 only, exact
manifest/profile fingerprint"; Call 3 requires `provenance_class=real` + c4 fingerprint). The
executing session has **`CLAUDE_CONFIG_DIR` unset**, so it runs on the default profile, not

which is the brief's own manifest-provenance hard stop.

This is the identical, already-documented reason the H963 campaign fired **zero** live calls
(H963 §"Why zero calls", reason 1). The condition is unchanged. The handoff's own guardrails
settle the ruling:

> A confirmed P0 that cannot be tested safely is a legitimate stop, not permission to spend
> around it.

> A supplied `GO` cannot override measured policy or `gate_reason`.

was measured about c4's health), not PRODUCTION_GO. Re-entry requires a session launched with
`CLAUDE_CONFIG_DIR` bound to the c4 profile — a human action, since it selects which account is
billed. Phases 1–5 are the prerequisites and are unaffected.
